**Project Context**: Apoio Pet (Spring Boot 4.0.7-SNAPSHOT + MySQL + JPA + data.sql seeding)  
**Date of Plan**: 2026-05-25  
**Related Recent Work**: Column name mismatch between entities (camelCase) and actual DB (snake_case) was identified and fixed. `data.sql` was corrected. Current state: `spring.sql.init.mode=never`, `ddl-auto=update`, `defer-datasource-initialization=true`.

## 1. Executive Summary

Even when `INSERT` statements are syntactically correct and "execute without error", rows can be absent from MySQL tables for several well-known reasons. In this Spring Boot project the most likely historical causes were:

1. **Column name mismatch** (already fixed) — `data.sql` referenced `emailAdm` / `nomeAnimal` while the live schema had `email_adm` / `nome_animal`.
2. **Initialization disabled** — `spring.sql.init.mode=never` (current intentional state).
3. **Timing / ordering** between Hibernate schema creation (`ddl-auto`) and script execution.

The user's new question focuses on **general and Spring-specific reasons** INSERTs can appear to succeed yet leave tables empty, plus a repeatable diagnostic + fix process.

## 2. General Reasons MySQL Tables Can Stay Empty After INSERT

### 2.1 Transaction / Commit Issues (Most Common "Silent" Cause)
- **Autocommit = false** on the connection used for the INSERT, followed by connection close without explicit `COMMIT`.
- Uncommitted transaction that is rolled back on connection close, client disconnect, or server crash.
- Implicit commit behavior of DDL mixed with DML inside the same script/session.
- Long-running transaction holding locks; another session sees old data or nothing.
- MySQL `REPEATABLE READ` isolation + non-repeatable phenomena within a transaction.

**Diagnostic**:
```sql
SHOW VARIABLES LIKE 'autocommit';
SHOW ENGINE INNODB STATUS\G
SELECT * FROM information_schema.INNODB_TRX;
```

### 2.2 Incorrect Database / Schema / Connection Selection
- JDBC URL points to `db_animal` but the user is connected to `test`, `information_schema`, or another MySQL instance.
- Multiple MySQL servers running (3306 vs 3307, XAMPP vs official installer).
- User is looking at the wrong schema in MySQL Workbench / DBeaver while the app uses a different catalog.

**Diagnostic**:
```sql
SELECT DATABASE();          -- must return 'db_animal'
SELECT @@hostname, @@port;
SHOW TABLES FROM db_animal;
```

### 2.3 Failed or Partially Failed Execution
- Script threw an exception but `spring.sql.init.continue-on-error=true` (or equivalent in manual run).
- Foreign key or unique constraint violation on later rows → earlier rows may have been rolled back depending on transaction.
- MySQL `max_allowed_packet` or statement timeout silently truncating multi-row INSERT.
- `data.sql` contains statements that are not pure DML (e.g., `USE`, `DELIMITER`, conditional logic) that the Spring `ResourceDatabasePopulator` does not understand.

### 2.4 Ordering & Lifecycle Issues (Spring Boot + Hibernate Specific)
- Hibernate `ddl-auto=create` or `create-drop` runs **after** the script and drops the just-inserted data.
- `spring.jpa.defer-datasource-initialization=false` (default) → script runs before Hibernate creates tables → FK violations or "table does not exist".
- `spring.sql.init.mode=embedded` (default) + production MySQL → script never runs.
- DevTools restart triggers multiple context refreshes; first one seeds, second one may overwrite or fail.

### 2.5 Other
- `sql.init.platform` mismatch or missing platform-specific script.
- Connection pool (Hikari) returning different physical connections; one connection sees the data, another does not (very rare, usually replication lag).
- Binary log / GTID issues in replicated setups.
- Application code using `@Transactional` with `propagation=REQUIRES_NEW` or wrong isolation that hides the data from the caller's view until commit.

## 3. Spring Boot 4 + JPA Data Initialization Specifics

Key classes (Spring Boot 4 / Spring Framework 7):
- `DataSourceScriptDatabaseInitializer`
- `AbstractScriptDatabaseInitializer`
- `ResourceDatabasePopulator` / `DatabasePopulatorUtils`
- Interaction with `HibernateJpaVendorAdapter` and `LocalContainerEntityManagerFactoryBean`

Important documented behaviors:
- Script initialization happens **before** JPA `EntityManagerFactory` by default.
- Set `spring.jpa.defer-datasource-initialization=true` to let Hibernate create tables first, then run `data.sql`.
- The populator obtains a connection from the `DataSource` (HikariCP in this project). Hikari default `autoCommit=true`.
- Historical bugs existed around `DatabasePopulatorUtils` not committing when `autoCommit=false` (see Spring Framework issue #27008).
- `spring.sql.init.mode=always` forces execution on any database (including MySQL).
- By default, any error in the script causes the entire application context to fail to start (fail-fast).

Current project settings (as of plan creation):
- `spring.sql.init.mode=never`
- `spring.jpa.hibernate.ddl-auto=update`
- `spring.jpa.defer-datasource-initialization=true`
- No custom `DataSourceInitializer` beans or `@Sql` annotations.

## 4. Project-Specific Root Cause Analysis (Apoio Pet)

1. **Primary historical cause**: Entity `@Column(name=...)` and `@JoinColumn(name=...)` used camelCase while Hibernate had already created snake_case columns in MySQL. `data.sql` INSERTs referenced non-existent columns → either startup crash or (when suppressed) zero rows.
2. **Current protective state**: `mode=never` prevents repeated failed or duplicate-key attempts.
3. **Risk if user blindly sets `mode=always` now**:
   - With `ddl-auto=update` the seed will run every restart → duplicate key errors on `tb_administrador.email_adm`, `tb_pelagem.pelagem`, etc.
   - With `ddl-auto=create` every restart wipes previous manual data.

## 5. Comprehensive Troubleshooting Guide (Iterative Diagnostic Tree)

### Level 1 – Quick Sanity (30 seconds)
1. In the running app or after attempting seed, execute in MySQL client:
   ```sql
   USE db_animal;
   SELECT DATABASE();
   SHOW TABLES;
   SELECT COUNT(*) FROM tb_administrador;
   SELECT COUNT(*) FROM tb_animal;
   ```
2. Check application logs (DEBUG level) for lines containing:
   - "DataSourceScriptDatabaseInitializer"
   - "Initializing database"
   - "Failed to execute SQL script"
   - "HikariPool" connection acquisition

### Level 2 – Script Execution Evidence
- Temporarily set `logging.level.org.springframework.jdbc=DEBUG` and `logging.level.org.springframework.boot.sql=DEBUG`.
- Look for the exact statements being executed by the populator.
- Check `spring.sql.init.continue-on-error` (should be false/unset for safety).

### Level 3 – Connection & Transaction Audit
- While the app is starting or right after a manual multi-statement INSERT:
  ```sql
  SELECT * FROM performance_schema.events_statements_current WHERE SQL_TEXT LIKE '%INSERT%';
  SHOW PROCESSLIST;
  SHOW ENGINE INNODB STATUS\G   -- look for "TRANSACTION" sections with "ACTIVE"
  ```
- From the app side, add a simple `@PostConstruct` bean (temporarily) that does `entityManager.createNativeQuery("SELECT 1").getSingleResult()` and logs the current DB.

### Level 4 – Isolation Test
- Manually run the exact `data.sql` content via the MySQL client (copy-paste) inside an explicit transaction:
  ```sql
  START TRANSACTION;
  -- paste all INSERTs
  COMMIT;
  ```
  Then immediately `SELECT COUNT(*)`. If rows appear only after `COMMIT`, the issue is autocommit/transaction handling in the Java side.

## 6. Step-by-Step Iterative Fix Plan (With Validation Gates)

**Goal**: Reach a state where `data.sql` reliably seeds a clean database once, then the app runs safely with `update` + `never` for normal development.

### Phase 0 – Preparation (Safe Snapshot)
- Backup current DB: `mysqldump -u root -p12345 --databases db_animal > backup_$(date +%F).sql`
- Note the exact current row counts for all tables.

### Phase 1 – Force Script Execution Path (Diagnostic Run)
1. Set temporarily:
   ```properties
   spring.jpa.hibernate.ddl-auto=create
   spring.sql.init.mode=always
   spring.sql.init.continue-on-error=false   # explicit
   ```
2. (Optional but recommended) Drop the database first for a truly clean run:
   ```sql
   DROP DATABASE IF EXISTS db_animal; CREATE DATABASE db_animal;
   ```
3. Start the app and capture **full** startup logs.
4. **Validation Gate**: After context refresh, immediately query row counts. Expected: 1 admin + 12 pelagens + 2 sexos + ... + 4 animals, etc.
5. If still zero rows:
   - Inspect logs for the exact failure (column names should now be correct).
   - Check `SELECT DATABASE()` from a `@PostConstruct` probe bean.
   - Try adding `spring.sql.init.platform=mysql` explicitly.

### Phase 2 – Isolate Transaction Behavior (If Phase 1 Still Fails)
1. Create a temporary manual test: a `@Component` with `@PostConstruct` that uses `JdbcTemplate` to run a single `INSERT` + `SELECT COUNT` in the same method.
2. If the manual `JdbcTemplate` INSERT is also invisible, the root cause is almost certainly the connection's autocommit or a surrounding transaction.
3. Mitigation options (choose one):
   - Explicit `COMMIT` via `JdbcTemplate.execute("COMMIT")` after the populator (hacky).
   - Configure a custom `DataSourceScriptDatabaseInitializer` bean that wraps the populator and forces commit (advanced).
   - Ensure Hikari `autoCommit: true` (default, verify in `application.properties` or via actuator).

### Phase 3 – Safe Production-Style Seeding Pattern
Once Phase 1 succeeds:
1. Immediately change back:
   ```properties
   spring.jpa.hibernate.ddl-auto=update
   spring.sql.init.mode=never
   ```
2. Restart the app multiple times.
3. **Validation Gate**: Row counts must remain stable (no duplicate key explosions, no data loss).

### Phase 4 – Handle the "I Want Fresh Data Every Restart" Desire
If the user wants clean data on every dev restart:
- Keep using `create` + `always` **only** when they explicitly want to reset.
- Or introduce a Spring profile `seed`:
  ```properties
  # application-seed.properties
  spring.jpa.hibernate.ddl-auto=create
  spring.sql.init.mode=always
  ```
  Run with `-Dspring.profiles.active=seed`.

### Phase 5 – Long-term Hardening (Recommended for Future)
- Migrate from raw `data.sql` to **Flyway** or **Liquibase** (versioned migrations + repeatable seed scripts). This is the industry-standard solution for exactly this class of problem.
- Or keep `data.sql` but guard it with `DELETE` + `INSERT ... ON DUPLICATE KEY UPDATE` (idempotent).
- Add integration tests using `@Sql` + H2 that do not depend on the production MySQL seeding path.
